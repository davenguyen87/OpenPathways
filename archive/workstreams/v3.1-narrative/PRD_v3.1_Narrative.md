# PRD v3.1: Audit Narrative Generation

**Status:** Shipped (2026-05-08). All acceptance criteria met; 16 unit + integration tests in `test/lib/audit-narrative.test.js`.
**Date:** 2026-05-08
**Author:** Dave Nguyen
**Built by:** Claude (autonomous)
**Supersedes nothing.** Extends [PRD v3](../PRD_v3_SkillLoop_Scoping.md) by adding LLM-generated narrative sections to the brand-matched HTML report, the Markdown variant, and the library-level rollup. The audit pipeline, scorecard schema, triage taxonomy, scope estimator, and existing programmatic report sections are unchanged. v3.1 adds (a) a narrative module, (b) a small reporter slot in each renderer, and (c) the audit-flow + CLI glue to opt in.

This is the second LLM-activation phase. v4.1 (assisted rebuild) was first; v5.x (transformer judgment) follows.

---

## Problem statement

v3 closed the gap between raw findings and consultant deliverable. The HTML report's Sections 01–08 cover the data layer (stats, triage rollups, scope estimates, top risks, per-package detail). What it does not cover is the *prose* layer: the executive narrative the consultant currently hand-writes ("This package has X violations clustered around alt text and form labels; remediation is dominated by author rework, not mechanical fixing"), the per-criterion guidance specific to *this* package's findings, and the prioritized scope memo that tells the buyer where to start.

For a single package the consultant can write that prose in 10–20 minutes. For a 200-package library it's the dominant cost in the engagement after audit itself. v3.1 generates first-draft narrative from the audit's own scorecard, places it in a clearly-labeled slot in the report with a "Generated by [model] — review before sharing" provenance pill, and lets the consultant edit before sending. The deliverable still requires human sign-off; it now requires editing, not authoring from scratch.

---

## Goals

1. **Cut consultant prose authoring time.** Replace blank-page authoring with editing of model-generated drafts that already cite the actual scorecard data.
2. **Hold the v3 deliverable contract.** The HTML report's Sections 01–08, the Markdown variant, the JSON scorecard, and the library rollup all remain authoritative. Narrative is additive — a new "Section 01a — Engagement Narrative" between the title block and Section 01.
3. **Make every narrative section auditable and reversible.** Each section carries provenance (provider, model, prompt hash, usage, latency, timestamp). When narrative is absent (LLM off, section failed), the existing programmatic templates render unchanged.
4. **Hold the no-telemetry posture.** Narrative calls go only to the configured provider, only when `--llm-provider` is set, and `scripts/check-no-network.js` continues to assert clean egress for tests using a fake provider.
5. **Engagement isolation, same as v4.1.** Provider is reinstantiated per audit; engagement ID recorded in every section's provenance.

## Non-goals

1. **No content rewriting of the existing sections.** Sections 01–08 stay as they are. Narrative does not summarize them — it adds a *separate*, prose-first reading of the same data.
2. **No multi-modal input.** Narrative reads the scorecard JSON only. It does not read the rebuilt package's HTML, take screenshots, or call vision-capable models.
3. **No automatic publishing.** The "Generated by [model] — review before sharing" pill is a contract, not a suggestion. Consultants edit before sending.
4. **No background batch of narrative for already-audited packages.** Narrative happens in the audit pipeline; re-running narrative against an existing scorecard is a v3.2 question.
5. **No web/cloud surface changes.** v3.1 is engine + CLI only. The hosted service can adopt narrative later by passing the same options through `audit()`.

## The narrative

### Section 01a in the rendered report

Single new section between the cover and Section 01. Three prose blocks, each with a provenance pill:

1. **Executive narrative** — one or two paragraphs in plain English: what's going on with this package, what dominates the findings, what the consultant should know in 30 seconds. Targets the buyer (LMS admin / training director), not the technical author.
2. **Per-criterion remediation guides** — one paragraph per failed criterion. What's wrong *in this package specifically* (not just the WCAG SC text), and what the fix shape is. Replaces the "rule definition" boilerplate the consultant currently writes by hand.
3. **Recommended remediation order** — a short prioritized list with rationale. "Start with the 23 missing-alt-text items because they unblock screen-reader navigation across every page; defer the contrast token rewrites until after the brand refresh you mentioned in scoping."

Each block is a separate prompt. They render in this order. If a block fails or is skipped, the others still render.

### Provenance pill

A small, brand-matched pill rendered immediately under each block's heading:

```
[ AI-DRAFTED · claude-haiku-4-5 · 2026-05-08T14:22Z · review before sharing ]
```

Renders in HTML using the existing brand palette's "system" color (a muted gray-teal). In Markdown, renders as italics. The pill is non-decorative — it is the documented contract that the consultant must read and edit the block.

### Library synthesis (rollup only)

When `--engagement <id>` runs across a directory (`audit-library`), one extra call: a library-level synthesis. Two paragraphs that describe patterns across packages: "47 of the 52 packages share the same Storyline 360 div-soup tab pattern; recommend a single template fix that lifts compliance for all of them." Renders at the top of `_library-rollup.{html,md}` with the same provenance pill.

## Output shape — `AuditNarrative`

```jsonc
{
  "schemaVersion": "1.0.0",
  "executive": {
    "text": "<plain-text or simple inline-markdown prose; renderer escapes / formats>",
    "provenance": {
      "source": "llm",
      "provider": "anthropic",
      "model": "claude-haiku-4-5",
      "promptHash": "sha256:…",
      "usage": { "inputTokens": 1240, "outputTokens": 180 },
      "latencyMs": 940,
      "generatedAt": "2026-05-08T19:22:11Z"
    }
  },
  "remediationGuides": [
    {
      "criterion": "1.1.1",
      "criterionName": "Non-text content",
      "text": "<one-paragraph guidance specific to this package's 1.1.1 findings>",
      "provenance": { … }
    }
    // … one entry per failed criterion that the model could ground in scorecard data
  ],
  "scopeMemo": {
    "text": "<short prioritized list with rationale>",
    "provenance": { … }
  },
  "totals": {
    "sectionsAttempted": 5,
    "sectionsSucceeded": 5,
    "totalInputTokens": 4120,
    "totalOutputTokens": 720,
    "totalLatencyMs": 3140
  }
}
```

`LibrarySynthesis` is a strict subset:

```jsonc
{
  "schemaVersion": "1.0.0",
  "synthesis": {
    "text": "<two-paragraph cross-package synthesis>",
    "provenance": { … }
  },
  "totals": { … }
}
```

Any field set to `null` means "section was skipped or failed; renderer falls back to its existing programmatic template (or simply omits, depending on the section)". Renderers must never crash on a missing or partial narrative.

## Prompt design

### Executive narrative

**Inputs sent.** `scorecard.summary` (totals + by-severity + by-confidence), `scorecard.criteria` (id, name, level, violationCount), `triage.rollup` (dominantTag, byTriage), `scopeEstimate` (hours bands), `topRisks` (top three), `engagementId`, `clientName` (when not `--redact-client-name`), `packageType`.

**System prompt.** Voice: Skill Loop accessibility consultant writing for the buyer (LMS admin / training director). Constraints: ≤ 240 words; two paragraphs; first paragraph names what dominates the findings (citing real numbers from the scorecard); second paragraph explains what the consultant would do first, in plain language. No jargon (no "WCAG 2.1 AA SC 1.4.3"); reference criteria by name ("color contrast"). No file paths, no SCO IDs. No invented numbers — every figure must trace to the scorecard inputs.

**Output validation.** Reject if: word count > 280, the text contains a number that does not appear in the scorecard inputs (basic numeric grounding check), the text contains a literal `<` or `>` (avoid raw HTML smuggling), or the text starts with `"As an AI"` or `"I am"`. On rejection, the section defers and renders the existing static "executive summary 4-stat grid" alone.

### Per-criterion remediation guides

One prompt per failed criterion, capped at 12 by default (`--llm-narrative-criterion-cap 12`). The cap exists because token budget — for a package with 30 failed criteria, narrative-generating all of them is wasteful; the top 12 by violation count cover the practical remediation surface.

**Inputs sent.** `criterion.id`, `criterion.name`, `criterion.url`, the violations on this criterion (file, line, message, snippet — clipped to first 5 examples to bound prompt size), the criterion's triage tag, this criterion's per-band effort estimate.

**System prompt.** Voice: senior accessibility consultant explaining the finding to a content author. Constraints: one paragraph, ≤ 100 words; describe *what's wrong in this package* (citing examples), then *what the fix shape is* (1–2 concrete actions, no quoted code). Do not paraphrase the WCAG SC text — assume the author can read the linked URL. No file paths in prose (the table of violations already shows them).

**Output validation.** Reject if: word count > 120, contains the literal SC number (we want plain language), or contains file paths matching `/[a-z0-9_-]+\.(html?|js|css)/i` (those belong in the violations table, not the prose).

### Recommended remediation order

**Inputs sent.** All failed criteria with violation count + triage tag + effort band, plus the top three risks.

**System prompt.** Voice: consultant writing the scope memo. Constraints: a short prioritized list (3–7 items); each item is one sentence, leading with the action verb ("Start with…", "Defer…", "Batch-fix…"); every item names a criterion (by name, not number) and gives a one-clause rationale ("…because it unblocks screen reader nav across every page"). No file paths. No "we recommend"; declarative voice.

**Output validation.** Reject if: list has < 3 or > 8 items, any item exceeds 35 words, or the model produces a different format than the prompt specified (e.g. paragraphs instead of a list).

### Library synthesis

**Inputs sent.** Per-package: package name, total violations, dominant criterion, dominant triage tag, scope hours band. Aggregated: cross-package criterion frequencies, cross-package triage rollup, total scope hours.

**System prompt.** Voice: consultant writing the engagement-level synthesis. Constraints: two paragraphs, ≤ 320 words total; first paragraph names the dominant cross-package pattern (citing how many packages exhibit it); second paragraph recommends one or two engagement-wide actions ("a single template fix lifts compliance across all 47 Storyline 360 packages").

**Output validation.** Same shape as executive narrative.

## Cost & safety guardrails

- **Per-package narrative budget.** Default `--llm-narrative-token-budget 30000` (~$0.04 / package on Haiku 4.5 across all sections). When the running total exceeds the budget, remaining sections skip with reason `narrative token budget exceeded`. Override with the flag.
- **Per-call timeouts and retries.** Reuse the v4.1 provider abstraction's defaults (15s timeout, one retry on 429/5xx). On persistent failure the section skips; the audit never errors because of narrative.
- **No fallback model.** If the configured model rate-limits past one retry, the section skips. We do not silently swap to a different model.
- **Engagement isolation.** Provider is reinstantiated per `audit()` call; engagement ID recorded in every section's provenance.
- **Numeric grounding check.** The executive and library-synthesis validators reject text containing numbers that don't appear in the scorecard inputs. This is a coarse check — it catches the most obvious hallucinations (made-up percentages, invented violation counts) without claiming to be a full fact-checker. The provenance pill remains the contract: consultants must read and edit before sending.

## CLI surface

```
prism audit <pkg.zip> --engagement <id>
  [--llm-provider anthropic|openai]
  [--llm-key-from-env <env-var-name>]
  [--llm-model <model-id>]
  [--no-llm-narrative]                      # opt out of narrative (default on when --llm-provider set)
  [--llm-narrative-token-budget <n>]        # default 30000
  [--llm-narrative-criterion-cap <n>]       # default 12

prism audit-library <dir> --engagement <id>
  [same flags as above]
```

`--llm-provider` and `--llm-key-from-env` already exist on `audit` (v3 scaffolding). v3.1 adds `--llm-model`, `--no-llm-narrative`, `--llm-narrative-token-budget`, and `--llm-narrative-criterion-cap` to both audit subcommands.

**Mode interactions.**

| `--llm-provider` set | `--llm-narrative` | `--no-llm-narrative` | Behavior |
|---|---|---|---|
| no | — | — | No LLM calls. Reports render exactly as today. |
| yes | (default) | — | Narrative auto-on. Sections render under Section 01a. |
| yes | — | passed | LLM is configured but narrative is off. Reserved for future LLM consumers (semantic checks, etc.). |
| yes (invalid env var) | any | any | `validateLlmConfig()` throws; CLI exits 2. |

## Acceptance criteria for v3.1

The release ships when all of the following are true:

1. `prism audit <fixture> --engagement test --llm-provider anthropic --llm-key-from-env ANTHROPIC_API_KEY` produces a `report.html` whose new "Section 01a — Engagement Narrative" contains all three blocks (executive, per-criterion guides, scope memo) with provenance pills.

4. `prism audit-library <dir> --engagement test --llm-provider …` produces `_library-rollup.html` with a "Library Synthesis" block at the top.
5. Every narrative section in the rendered HTML has the brand-matched provenance pill containing provider, model, ISO timestamp, and "review before sharing".
6. `results.json` carries an `auditNarrative` object alongside the existing scorecard fields when narrative ran (and omits the field cleanly when it didn't).
7. `npm test` passes. The integration test injects a fake provider; no real network call in CI.
8. `npm run check-no-network` passes. The fake-provider test path does not trigger the egress trap.
9. Reports without narrative remain byte-identical to v3 output (regression test).

## Risks and mitigations

- **Hallucinated numbers in executive prose.** The numeric grounding validator rejects text containing numbers absent from the scorecard inputs. The provenance pill remains the durable contract. Worst case: a single bad section the consultant deletes during editing.
- **Narrative pushes the consultant toward over-trusting the model.** Mitigation: the pill is mandatory, not stylistic; the documentation in `docs/CONTRACT.md` (to be updated) names narrative as draft, not deliverable. Internal training material should reinforce.
- **Per-criterion guides leak SC paraphrases that conflict with the linked WCAG text.** The validator rejects text containing the SC number; the prompt instructs against paraphrasing. If the model still generates conflicting prose, the consultant edits or deletes the block.
- **Token budget runs away on large libraries.** Default budget plus per-section caps bound the worst case. The library rollup also tracks aggregate narrative token spend so the consultant sees the cost.
- **A model deprecation breaks reproducibility of narrative.** The provenance records the model id at generation time. Re-running `--llm-provider …` with `--llm-model <new-id>` replaces the narrative; the diff is visible in the report.
- **Brand drift between programmatic sections and narrative.** Narrative is rendered using the same brand palette and typeface stack as Sections 01–08. The provenance pill is the only new visual element; it uses the existing system-color token rather than a new color.
